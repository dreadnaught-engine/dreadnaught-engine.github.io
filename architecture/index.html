Dreadnaught Manifesto

The foundational engineering constraints, technical stack, and data-oriented rules governing the Dreadnaught Engine.

1. Tech Stack & Major Decisions

  • Language & Design Philosophy: C++23 with strict DOD/ECS. Modern features like this auto, if consteval, and std::mdspan are leveraged specifically to write clean, compile-time-optimized code.
  • Graphics & Pipelining: A stateless, explicit Render Hardware Interface (RHI) utilizing Bindless Rendering and Mesh Shaders.
  • Physics & Kinematics: Jolt Physics handles global rigidbodies. VR instances utilize High-Fidelity Active Ragdolls; 60-player raids enforce Custom Kinematics to preserve server budgets.
  • Build System: Modern CMake. All third-party libraries use strict vendoring / Git Submodules to guarantee 100% offline compilation and build determinism.

2. Architectural Pillars

Pillar 1: Tiered Epoch Allocators

Memory is scoped hierarchically (Persistent Data -> Instance Base Data -> Procedural Phase Data). Procedural Phases utilize a Ping-Pong Arena topology to prevent main-thread stutter while waiting for asynchronous callbacks to unmap memory.

Pillar 2: Headless Server Compilation

The ECS, Physics, and Game Logic are strictly decoupled from Graphics and Audio via preprocessor definitions. The engine treats single-player natively as a local Listen Server.

Pillar 3: Unified Hybrid Networking

Snapshot Interpolation & Delta Compression is utilized for remote entities. The server strictly sends mathematical deltas. The client synchronizes via Phase-Locked Loop (PLL) Time Dilation.

CRITICAL RULE: "Double-Ticking" the ECS within a single render frame is strictly forbidden to prevent multithreaded thread-barrier deadlocks.

Pillar 4: ECS Anti-Fragmentation

Mutable entity states utilize Pre-Allocated Fixed-Size Buffers (e.g., AbilityBuffer). Rapidly adding/removing ECS components is forbidden. Systems overwrite hashes within buffers to guarantee the entity remains in its original Archetype memory chunk.

Pillar 7: Data-Oriented ECS & Transforms

Entities are lightweight IDs. Components are pure POD structs stored in an SoA layout. To support hierarchies without OOP pointers, we use a ParentLink component and a vectorized Transform Resolution System.

Pillar 12: Strict Core Allocator Discipline

FORBIDDEN: Zero hidden allocations in the core. Standard library containers (std::vector, std::unordered_map, std::string) are banned. Runtime string execution is strictly forbidden.

The engine mandates an EASTL-style custom container library. Strings must be hashed into 64-bit integers via constexpr at compile-time. Asset dependencies rely purely on permanent, 64-bit intrinsic GUIDs.

Pillar 14: Asynchronous Direct-to-GPU Streaming

To guarantee seamless portal transitions, a background fiber pipeline leverages DirectStorage to stream upcoming procedural chunks directly from SSD to GPU VRAM, bypassing the CPU main thread entirely.

3. Coding Constraints & Standards

POD MANDATE: Components are strictly Plain Old Data (POD). Fixed-size arrays (std::array) are permitted only if the maximum size is small and uniform.
  • Strict Determinism: Standard stateful RNG (std::mt19937, rand()) is forbidden. Worker fibers must utilize explicitly seeded, fiber-local RNG states synced to the server's simulation frame.
  • SIMD Memory Alignment: Core engine math is natively aliased to 16-byte Jolt Physics SIMD structures. Any ECS component containing spatial vectors MUST utilize the alignas(16) keyword.
  • Network Serialization: We strictly forbid OOP-based serializers (like Google Protobuf) that rely on hidden heap allocations. All payloads must be pure POD byte-streams pushed directly to a pure-C UDP layer.
